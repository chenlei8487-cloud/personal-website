Replacing a SaaS Approval Tool with Power Automate — at Zero Cost

IKEA China · Process Automation · 2026

Power Automate SharePoint Microsoft Forms Process Automation Workflow

The Problem

When the company cancelled its SaaS approval tool for cost reasons, the Property team was left with no structured approval system. Teams fell back to email-based approvals — and the cracks showed immediately:

  • Requests got lost. Approval emails mixed with daily inbox traffic. Important decisions sat unnoticed.
  • No audit trail. Approval records scattered across personal mailboxes. A compliance review meant hours of inbox archaeology.
  • No visibility. Requestors had no idea where their submission was stuck or who needed to act.
  • Resubmission was painful. Rejected requests meant back-and-forth emails with no version control.
  • No structured data. No way to generate reports, track cycle times, or measure team performance.

The business need was clear. The budget for a replacement tool was zero. I built one.

The Solution

Using only existing Microsoft 365 licenses, I designed and built a multi-level approval system in two days. The architecture has three components:

1. Entry Point: Microsoft Forms

Anyone can submit a request through a standardized form. They enter the project, job type, description, amount, and upload supporting documents.

2. Workflow #1: Data Processing + ID Generation (Power Automate)

When a form is submitted, the first flow runs immediately:

  • Generates a unique request ID based on job type and project (e.g., RFI-PV-011)
  • Converts attached files into HTML links for SharePoint display and Markdown links for email notifications
  • Initializes metadata — status, timestamps, requestor identity
  • Writes everything to a SharePoint list (Request Log)

3. Workflow #2: Multi-Level Approval (Power Automate)

Creating a new Request Log entry triggers the approval flow:

  • Looks up the Approval Matrix — a configurable table that defines who approves what, based on job type and amount range
  • Runs through approval levels sequentially (L1 → L2 → L3), with "all must approve" at each level
  • Updates the Request Log in real time — requestors can see exactly who is currently reviewing
  • Sends a notification email for each terminal state: Approved, Rejected, or Flow Failed

Behind the Scenes: Two Configuration Tables

Approval Matrix — defines routing rules. Each row specifies: job type, amount range, and which roles must approve at each level. For example, an RFI under $10,000 goes to Manager → Director, while one over $50,000 goes to VP → CFO → CEO.

Approver Mapping — decouples roles from people. When a team member moves projects or leaves, an admin updates one row in this table. The approval flows keep running without any code changes.

Results

Metric Before After
Average approval cycle 5–7 days 2–3 days
Request omission rate ~15% (estimated) 0% (system-tracked)
Archive search time 30+ minutes <1 minute
New hire ramp-up Verbal walk-through required Self-guided with documented process

Additional highlights:

  • $0 development cost — built entirely on existing Microsoft 365 licenses
  • 2 days from requirements to working system
  • Zero migration friction — team only needed to learn one Microsoft Form; everything else happened automatically

What This Demonstrates

  1. System design thinking. The two-flow architecture separates concerns: data intake (Flow #1) is independent from routing logic (Flow #2). A change to either side doesn't break the other.
  2. Configuration over code. Approval rules live in SharePoint tables, not buried in Power Automate expressions. Non-technical administrators can update routing without touching a flow.
  3. User experience matters even in back-end tools. Email notifications with clear status, rejection reasons, and next steps reduced follow-up questions to zero.
  4. Constraints breed creativity. Zero budget meant no new tools to learn, no procurement process, no vendor calls. Just existing infrastructure used smarter.
